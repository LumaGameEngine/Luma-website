The Manifesto

Software Design Over Hardware Bloat

Most engines assume infinite hardware. We assume you have a brain. Luma Engine is built for developers who understand that great software is crafted, not compiled into existence.

"Great software isn't built by demanding faster hardware. Great software is built by making better use of the hardware we already have."
— Princeflouz, Lead Architect
@Princeflouz Discord

Cache-First

Data structures optimized for L1/L2 cache hits. No stutter.

Data-Oriented ECS

Hybrid entity-component system. Performance predictable.

Custom Allocators

No dynamic allocation during gameplay. Deterministic performance.

UUID-Based Assets

128-bit UUIDs for every asset. Rename without breaking references.

Core Rendering

Dimension_L DL-V1

Render only what matters. Three visual tiers, one goal: efficiency.

Tier 1

Near Layer

Status: Full Quality
  • Full vector pipelines
  • Per-pixel lighting
  • Full matrix updates
Tier 2

Mid Layer

Status: Cached
  • Interpolated geometry
  • Pre-rendered buffers
  • Static layer skipping
Tier 3

Far Layer

Status: Proxy
  • Flat texture proxies
  • No transform overhead
  • Minimal compute
Render what matters. Skip the rest.

Game Logic

The LOVE Language

A lightweight, sandboxed scripting language for gameplay logic. LOVE handles dynamic updates without exposing core engine memory.

Sandboxed

Gameplay runs in isolated scopes. Safe and secure.

Fast

Tokens resolve through sequential memory. Predictable performance.

Familiar

Feels natural for Lua, Python, and JavaScript developers.

player.love Syntax Preview

entity Player {

position = (0, 0)

health = 100


on_update(dt) {

if key("W") {

position.y -= 1

}

if key("A") {

position.x -= 1

}

}


on_draw() {

draw_sprite("player.png", position)

}

}

// LOVE — compiled to native C++ bytecode
